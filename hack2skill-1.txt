# FairLens — AI Bias Audit & Mitigation Platform
## Final Implementation Plan

---

## 1. Product Vision

An accessible, end-to-end web platform where any organization — bank, NGO, government department, HR team — can audit AI systems and datasets for hidden bias, receive plain-language explanations of what's unfair and why, and get actionable guidance to fix it. No Python skills. No data science degree. Upload, understand, act.

### Who Uses It

| User Type | Example | How They Use It |
|---|---|---|
| **Technical** | ML engineer at a fintech | Exports test predictions, runs full 6-metric audit, uses SHAP attribution to find root cause |
| **Semi-technical** | HR manager at a large company | Exports hiring decision log from ATS (Workday/Greenhouse), uploads Excel, reads Gemini report |
| **Non-technical** | NGO program coordinator | Types aggregate approval numbers directly into the UI, gets instant fairness verdict |
| **Procurement / Compliance** | Bank evaluating a 3rd-party credit scoring vendor | Probes vendor API with counterfactual test cases, generates compliance PDF |

## 2. Data Ingestion Modes

This is the critical innovation. Six distinct ways to get data into the audit engine — meeting every user where they are.

### Mode 1 — Prediction Audit (Structured CSV/Excel)
**Input:** CSV or Excel with columns for y_true (actual outcome), y_pred (model's decision), and demographic columns  
**What we audit:** Full model fairness — all 6 metrics  
**Target user:** ML engineers, data scientists  
**Processing:** Direct column mapping via interactive UI dropdowns  

### Mode 2 — Raw Operational Data Upload
**Input:** Any existing Excel/CSV the organization already uses — hiring tracker, loan disbursement sheet, beneficiary register  
**What we audit:** Historical bias in the data itself (representation, label skew, proxy features)  
**Target user:** HR managers, loan officers, NGO coordinators  
**Processing:** Smart column detection + interactive mapping wizard. User sees their actual column names and assigns roles: "Which column is the final decision?" "Which columns describe the person?"  

### Mode 3 — Dataset Readiness Check (Pre-Training)
**Input:** Raw dataset intended for AI training — features + outcome column, no separate predictions  
**What we audit:** Is this data safe to train on? Representation balance, historical label bias, proxy feature contamination  
**Target user:** Data scientists before training begins  
**Processing:** Same as Mode 2, but audit output explicitly answers "Will an AI trained on this data inherit bias?"  

### Mode 4 — Aggregate Input (No File Upload)
**Input:** User types summary numbers directly into a form:
```
Group A: 400 total, 340 approved
Group B: 300 total, 180 approved
```
**What we audit:** Demographic parity + disparate impact ratio (mathematically valid from group totals alone)  
**Target user:** Non-technical users who can read numbers off their screen but cannot export any file  
**Processing:** No file parsing — direct computation from form inputs  

### Mode 5 — API Probe Audit (Counterfactual Testing)
**Input:** User provides a model's REST API endpoint + sample input schema  
**What we audit:** Direct discrimination via counterfactual pairs. We generate identical profiles that differ only in the protected attribute (Male vs Female, Urban vs Rural) and observe if the model's response changes  
**Target user:** Procurement teams evaluating 3rd-party AI vendors  
**Processing:** Synthetic test case generator → batch API calls → response comparison → fairness metrics on the synthetic audit  

### Mode 6 — Document / Screenshot Upload
**Input:** PDF decision reports, screenshots of AI tool interfaces, forwarded decision emails  
**What we audit:** Whatever structured data Gemini Vision + OCR can extract  
**Target user:** Government departments, hospitals, anyone with locked-down systems that have no export function  
**Processing:** Gemini Vision API extracts tabular data → structured into internal format → same audit engine

## 3. Data Quality Gate (Runs Before Every Audit)

Immediately after data ingestion and before the audit starts, a validation layer runs and presents the user with a traffic-light report:

### Checks Performed

| Check | What It Catches | User Sees |
|---|---|---|
| **Group size per attribute** | Underrepresented demographics | ✅ "Male: 423 samples" / ⚠️ "Non-binary: 12 samples — results unreliable" |
| **Outcome balance** | Extreme class imbalance | ⚠️ "Only 3% positive outcomes — consider using Equal Opportunity over Demographic Parity" |
| **Missing values per column** | Data holes | "Gender: 8 missing → these rows excluded" |
| **Single-value columns** | Cannot audit — no comparison possible | ❌ "Column 'Country' has only one value — skipping" |
| **Continuous protected attribute detection** | Age/income as raw numbers | Auto-suggestion: "Age detected as continuous — we'll bucket into 18-25, 26-35, 36-50, 50+" |
| **y_true == y_pred check** | User uploaded dataset mode as prediction mode | ⚠️ "Predictions match ground truth exactly — did you mean Dataset Mode?" |
| **Outcome column = protected attribute overlap** | Invalid configuration | ❌ "Outcome column cannot also be a protected attribute" |
| **Minimum total rows** | Dataset too small for any reliable analysis | ❌ "Only 15 rows — minimum 50 required for meaningful audit" |

User sees this as a clean summary screen with ✅ ⚠️ ❌ indicators. They can proceed with acknowledged limitations or go back to fix their data.

## 4. Fairness Audit Engine

### 4.1 Core Metrics (Per Protected Attribute)

| # | Metric | Formula (Intuitive) | Fail Threshold | What it Means in Plain English |
|---|---|---|---|---|
| 1 | **Demographic Parity Difference** | P(approved \| Group A) - P(approved \| Group B) | \|diff\| > 0.10 | "Are approval rates equal across groups?" |
| 2 | **Equal Opportunity Difference** | P(approved \| actually qualified, A) - P(approved \| actually qualified, B) | \|diff\| > 0.10 | "Among people who deserved approval, does the model miss one group more?" |
| 3 | **Disparate Impact Ratio** | min(group rate) / max(group rate) | ratio < 0.80 | "The 80% rule — is the worst-off group getting less than 80% of the best-off group's rate?" |
| 4 | **Predictive Parity Difference** | Precision(A) - Precision(B) | \|diff\| > 0.10 | "When the model says 'approved', is it equally correct for all groups?" |
| 5 | **Overall Accuracy Equity** | Accuracy(A) - Accuracy(B) | \|diff\| > 0.05 | "Is the model equally accurate for all groups?" |
| 6 | **False Negative Rate Disparity** | FNR(A) - FNR(B) | \|diff\| > 0.10 | "Does the model wrongly reject qualified people more from one group?" |

Metrics 2, 4, 5, 6 require both y_true and y_pred (Prediction Mode only).  
Metrics 1 and 3 can be computed from outcomes alone (Dataset Mode) or from aggregate inputs (Mode 4).

### 4.2 Intersectional Analysis

After running single-attribute audits, automatically run **2-way cross-attribute audits** for every pair of protected attributes:
- Gender × Region
- Age × Income bracket
- Caste category × Gender

This catches cases where single-attribute audits pass but compound groups are discriminated. Example: "Gender is fair. Region is fair. But Rural Women are approved 42% less than Urban Men."

**Guard rail:** Cap at 2-way intersections. If a cross-group has < 30 samples, flag as unreliable but still show results.

### 4.3 Proxy Feature Scanner

For every non-protected feature column in the dataset, compute correlation with each protected attribute:
- **Cramér's V** for categorical × categorical
- **Point-biserial correlation** for categorical × numeric
- **Pearson correlation** for numeric × numeric

**Flag threshold:** correlation > 0.30 triggers a warning card:
> ⚠️ "Feature 'Pin Code' has 0.47 correlation with 'Caste Category' — this feature may act as a proxy for a protected attribute. Even if you remove caste from direct features, the model can learn caste through pin code."

### 4.4 SHAP Feature Attribution (Prediction Mode + Model Available)

When bias is detected and a model object or prediction probabilities are available, use SHAP to identify which input features drive the disparity between groups.

Output: A ranked list per flagged attribute:
```
Bias against Rural applicants is driven by:
  1. Distance to nearest branch: 38% contribution
  2. Monthly transaction count: 22% contribution
  3. Internet banking usage: 14% contribution
```

This transforms "bias detected" into "here's the specific cause."

**Library:** `shap` (TreeExplainer for tree models, KernelExplainer as fallback).  
**Fallback:** If no model object available (prediction-only CSV), skip SHAP and note: "Upload your model .pkl file to enable root cause analysis."

## 5. Gemini Integration Layer

### 5.1 Plain-Language Report Generation

**System Prompt:**
```
You are a fairness analyst explaining AI bias audit results to business 
decision-makers. Your job is to translate technical fairness metrics into 
clear, specific, actionable language.

Rules:
- Never use terms like 'confusion matrix', 'TPR', 'FPR', 'precision', 
  'recall', or 'statistical significance'
- Instead say 'approval rate', 'correctly identified', 'wrongly rejected', 
  'treated fairly'
- Always name the specific groups, the specific gaps, and the real-world 
  consequence
- Use concrete numbers: "34% fewer" not "significantly fewer"
- If a metric passes, say so explicitly — don't only report failures
- Structure: What the model does → What was found → Why it matters → 
  What to fix first
```

**User Prompt Template:**
```
Audit Results for {org_name}'s {model_name}:

Domain: {domain} (e.g., Hiring, Lending, Insurance, Government Scheme)
Audit Mode: {mode} (Prediction Audit / Dataset Audit / API Probe)
Total records audited: {total_rows}

Per-Attribute Results:
{audit_json}

Intersectional Results:
{intersectional_json}

Proxy Features Detected:
{proxy_json}

Generate:
1. EXECUTIVE SUMMARY (3 paragraphs, no jargon): What the system does. 
   What bias was found. Why it matters for {domain}.
2. ATTRIBUTE-BY-ATTRIBUTE BREAKDOWN: One paragraph per protected attribute 
   that failed, with specific numbers and named groups.
3. INTERSECTIONAL FINDINGS: Any compound discrimination discovered.
4. PROXY WARNINGS: Plain-language explanation of each proxy feature detected.
5. PRIORITY ACTION: The single most important fix and why it should be first.

Tone: Clear, direct, non-alarming but honest. Written for a 
{user_role_description} who has never seen a fairness metric before.
```

### 5.2 Gemini Vision — Document/Screenshot Extraction

For Mode 6 uploads, use `gemini-1.5-flash` with vision capability:

```
Prompt: "Extract all tabular data from this image/document. 
Return as a JSON array of objects where each object is one row. 
Identify columns that appear to be: person identifiers, 
demographic attributes, and decisions/outcomes."
```

### 5.3 Counterfactual Test Case Generation (Mode 5)

```
Prompt: "Given this API input schema: {schema}
Generate 50 pairs of test cases. Each pair must be identical 
in every field EXCEPT the protected attribute '{attr}'. 
One case should have value '{group_a}' and the paired case 
should have value '{group_b}'. All other fields should have 
realistic values for the domain: {domain}.
Return as JSON array."
```

## 6. Mitigation Engine

### 6.1 Triggered Mitigation Cards

Each card is triggered by a specific failure pattern. Cards are ranked by severity (Critical → Warning → Info).

#### Dataset Mode Cards

| Condition | Card Title | Severity | Action |
|---|---|---|---|
| Group size < 10% of total | "Underrepresented Group: {group}" | Critical | Collect more data from this group; use SMOTE synthetic oversampling |
| Outcome rate for group < 40% of best group | "Historical Bias in Labels" | Critical | Re-weight training samples; consider rewriting labels with domain expert review |
| Proxy feature detected (correlation > 0.3) | "Proxy Feature: {feature} → {protected_attr}" | Warning | Remove or transform this feature before training; use adversarial debiasing |
| Outcome imbalance > 90/10 | "Extreme Class Imbalance" | Warning | Apply class weights or resample before training |

#### Prediction Mode Cards

| Failing Metric(s) | Card Title | Severity | Action |
|---|---|---|---|
| Demographic parity fails | "Unequal Approval Rates" | Critical | Post-processing: adjust decision thresholds per group to equalize rates |
| Equal opportunity fails | "Missing Qualified Applicants from {group}" | Critical | Re-weight training samples — up-weight false negatives from disadvantaged group |
| Disparate impact < 0.80 | "Below Legal Fairness Threshold" | Critical | Remove proxy features, re-train with fairness constraints (Fairlearn ExponentiatedGradient) |
| Predictive parity fails | "Model Accuracy Differs by Group" | Warning | Train per-group calibrators (isotonic regression) |
| FNR disparity fails | "Wrongly Rejecting {group} at Higher Rate" | Critical | Threshold adjustment or re-training with group-aware loss function |
| Multiple metrics fail | "Systematic Bias Detected" | Critical | Full fairness-aware re-training with ExponentiatedGradient and DemographicParity constraint |
| Intersectional fails but single-attr passes | "Hidden Intersectional Blind Spot" | Warning | Stratify training data by the intersectional subgroup; audit that subgroup explicitly |

### 6.2 Trade-Off Simulator

When a mitigation is suggested, show the user the cost:

```
Current State:
  Overall accuracy: 92.1%
  Disparate Impact Ratio (Gender): 0.62 ❌

If you apply "Threshold Adjustment":
  Projected accuracy: 90.4% (↓1.7%)
  Projected Disparate Impact: 0.83 ✅

If you apply "Re-training with Fairness Constraints":
  Projected accuracy: 89.8% (↓2.3%)
  Projected Disparate Impact: 0.91 ✅
```

Implementation: Use Fairlearn's `ThresholdOptimizer` and `ExponentiatedGradient` with `GridSearch` to compute the Pareto frontier of accuracy-vs-fairness trade-offs. Display as a chart with a slider so the user can explore the trade-off.

> [!IMPORTANT]
> The mathematical impossibility result (Chouldechova / Kleinberg): You **cannot** simultaneously achieve demographic parity, equal opportunity, and predictive parity unless base rates are equal across groups. The report must explicitly explain this trade-off when multiple metrics conflict. This is where our Gemini explanation adds real value — translating a mathematical impossibility into a business decision.

## 7. PDF Compliance Report

### Structure

```
Page 1: Cover
  - FairLens Logo
  - Organization name
  - Model/System name  
  - Audit date
  - "AI Fairness Audit Report"

Page 2: Executive Summary
  - Gemini-generated plain-language summary (3 paragraphs)
  - Overall fairness verdict: PASS / PARTIAL FAIL / FAIL (with color badge)

Page 3-N: Per-Attribute Detail
  - For each audited attribute:
    - Metric table with pass/fail badges
    - Group-level bar chart (approval rates per group)
    - Gemini paragraph explaining this attribute's findings

Page N+1: Intersectional Findings
  - Cross-attribute results table
  - Highlighted compound discrimination cases

Page N+2: Proxy Feature Warnings
  - Correlation matrix heatmap
  - Plain-language proxy explanations

Page N+3: Mitigation Recommendations
  - Priority-ranked cards with actions
  - Trade-off projections if applicable

Page N+4: Methodology Statement
  - Metrics used and their thresholds
  - Data quality summary (rows, groups, exclusions)
  - Limitations and caveats
  - "Generated by FairLens — AI Bias Audit Platform"
```

**Technology:** WeasyPrint (HTML+CSS template → PDF). Charts rendered as inline SVGs using matplotlib.

## 11. Database Schema

```sql
-- Audit jobs table
CREATE TABLE audit_jobs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    org_name VARCHAR(255),
    model_name VARCHAR(255),
    domain VARCHAR(100),                    -- hiring, lending, insurance, government
    mode VARCHAR(50) NOT NULL,              -- prediction, dataset, aggregate, api_probe, document
    status VARCHAR(50) DEFAULT 'created',   -- created, configured, validating, running, complete, failed
    file_path TEXT,                         -- path to uploaded file (null for aggregate mode)
    config JSONB,                           -- audit configuration (columns, attributes, favorable outcome)
    quality_report JSONB,                   -- data quality gate results
    audit_results JSONB,                    -- per-attribute metric results
    intersectional_results JSONB,           -- cross-attribute results
    proxy_results JSONB,                    -- proxy feature scan results
    shap_results JSONB,                     -- SHAP attribution results
    gemini_report JSONB,                    -- plain-language report sections
    mitigation_cards JSONB,                 -- generated mitigation recommendations
    pdf_path TEXT,                          -- path to generated PDF
    total_rows INTEGER,
    created_at TIMESTAMP DEFAULT NOW(),
    completed_at TIMESTAMP,
    error_message TEXT
```

```python
# backend/app/constants.py

# ─── Fairness Metric Thresholds ──────────────────────────
DEMOGRAPHIC_PARITY_THRESHOLD = 0.10      # |difference| > this = fail
EQUAL_OPPORTUNITY_THRESHOLD = 0.10
DISPARATE_IMPACT_THRESHOLD = 0.80        # ratio < this = fail (80% rule)
PREDICTIVE_PARITY_THRESHOLD = 0.10
ACCURACY_EQUITY_THRESHOLD = 0.05
FNR_DISPARITY_THRESHOLD = 0.10

# ─── Proxy Feature Detection ────────────────────────────
PROXY_CORRELATION_THRESHOLD = 0.30       # Cramér's V or Pearson > this = proxy warning

# ─── Data Quality Minimums ───────────────────────────────
MIN_GROUP_SIZE = 30                      # Below this → "unreliable" warning
MIN_TOTAL_ROWS = 50                      # Below this → refuse to audit
MAX_INTERSECTIONAL_GROUPS = 20           # Cap cross-attribute group explosion
```

## 21. Risk Mitigations

| Risk | Mitigation |
|---|---|
| Fairlearn API breaking changes | Pin `fairlearn==0.10.0` |
| Gemini rate limit during demo | Cache responses in Redis; pre-generate for all 3 sample datasets |
| WeasyPrint system dependencies on Cloud Run | Use `python:3.11-slim` + install `libpango1.0-dev libcairo2-dev` in Dockerfile |
| Large CSV upload timeout | 50MB limit; async processing with background task + polling; show progress |
| Intersectional group explosion | Cap at 2-way intersections; cap at 20 unique groups; warn user |
| Gemini returns jargon despite prompt | Post-process: regex scan for banned terms, re-prompt if detected |
| API probe target rate-limits our calls | Configurable delay between calls; cap at 200 total calls per probe |
| Vendor API requires auth we can't handle | Support Bearer token, API key, and basic auth; document limitations |
