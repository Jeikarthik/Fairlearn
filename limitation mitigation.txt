Here is the exhaustive, brutally honest failure analysis of the FairLens system, categorized from "Garbage In" to "System Crashes".
Part 1: Every Possible Failure, Error, or Hallucination
A. Data Ingestion Failures (The "Garbage In" Problem)
The Dirty String Error: The CSV contains "Female", "female", "F", and " Woman". The system treats them as 4 different groups.
The Continuous Outcome Error: The model prediction is a Credit Score (300-800), not a Yes/No. The binary fairness math cannot compute.
The File Bomb: A user uploads a 4GB CSV file or a .txt file pretending to be a .csv, crashing the Pandas processing engine and maxing out Cloud Run memory.
The "Missing Context" Error: The user uploads data where the outcome is "0", but we don't know if "0" means "Hired" or "Fired".
Extreme Class Imbalance: The data has 99.5% approvals. Standard fairness metrics technically "pass" but are statistically meaningless.
B. Mathematical & Statistical Failures (The "Math Breaks" Problem) 6. Division by Zero: The disadvantaged group has 0 approvals. Disparate Impact Ratio tries to compute 0 / X or X / 0 and the Python backend throws a fatal exception. 7. Small Sample Noise (False Positives): A specific demographic group only has 8 people. 6 get rejected. The math claims an 80% bias rate. 8. Inherently Biased Ground Truth: The user's historical y_true data was racist. We benchmark the model against it, and declare the model "fair" because it perfectly matches the racist history. 9. Conflicting Metrics: The system fails Demographic Parity but passes Predictive Parity. The user feels the tool is broken or contradictory.
C. Generative AI Hallucinations (The "AI Lies" Problem) 10. The Number Hallucination: Fairlearn outputs a 22% gap. Gemini writes a beautiful paragraph claiming a huge 49% gap. 11. The Moral Context Hallucination: The data is for Hospital ICU Triaging (where rejection means death). Gemini frames it playfully or misinterprets the context as a loan approval. 12. The Jargon Dump: Gemini ignores the prompt and talks about "False Positive Rate Disparity and Confusion Matrices," confusing the HR manager. 13. Synthetic Data Failure (API Probe): In Mode 5, Gemini is told to generate a synthetic human profile to test an external API. It generates an age of -14 or an income of "Apple", causing the external API to instantly reject the test.
D. Infrastructure Failures (The "Demo Crashes" Problem) 14. The Cloud Timeout: The math takes 90 seconds. The HTTP browser request times out at 30 seconds. 15. The Gemini Quota Wall: During the live demo, we hit the Gemini "Requests Per Minute" quota limit. The UI spins forever or crashes. 16. API Probe Rate Limits: The 3rd-party vendor API detects us sending 50 rapid-fire requests and IP-bans us mid-audit using rate limiters (HTTP 429).

Part 2: Failures We ALREADY Tackle in the Implementation Plan
Because of the first plan and the Addendum we wrote earlier, our architecture is already highly resilient. Here is what we already have solved:
Tackles #4 (Missing Context): UI forces the user to explicitly define the domain (hiring, lending) and select the favorable_outcome value.
Tackles #5 (Extreme Class Imbalance): The Data Quality Gate runs before math, detecting <10% representations and warning the user.
Tackles #7 (Small Sample Noise): Addendum Change #4 (Wilson Confidence Intervals). Metrics fallback to an "Inconclusive" state if sample size breeds uncertainty.
Tackles #8 (Inherently Biased Ground Truth): Addendum Change #3 (Ground Truth Health Check). We audit the history itself before auditing the model.
Tackles #9 (Conflicting Metrics): Addendum Change #5 (Trade-off Panel). UI explicitly explains the mathematical impossibility of satisfying all metrics simultaneously.
Tackles #10 (Number Hallucination): Addendum Change #7 (Gemini Post-Validation). Backend intercepts text, regex-extracts numbers, and cross-checks the JSON. Reprompts if Gemini lied.
Tackles #11 & #12 (Moral Context & Jargon Dump): Addendum Change #7 (Banned Word Scanner + Domain Prompting).
Tackles #13 (Synthetic Data Failure): Mode 5 utilizes Gemini "Structured Output" (Pydantic schemas) to force the LLM to output valid JSON matching the vendor APIs.
Tackles #14 (Cloud Timeout): The API contract uses Async Background Tasks (status: running) and frontend polling (useAuditPolling.js).
Tackles #16 (API Probe Rate Limits): Addendum Change #9 adds exponential backoff arrays to API endpoints to dodge HTTP 429 Bans.

Part 3: Strategies We MUST Include for the Remaining Errors
We need an "Addendum v2" to add these specific structural fixes to the plan:
Strategy 1: The String Normalizer Pipeline (Fixes #1 - The Dirty String Error)
Implementation: Before the Data Quality Gate, add a clean_categorical_data() pandas function. It executes .str.strip().str.lower().str.title() to merge female, Female , and F together via a standard mapping dictionary, surfacing a warning if too many unique anomalies remain.
Strategy 2: The Continuous Outcome Converter (Fixes #2 - The Continuous Outcome Error)
Implementation: In the Configuration UI, if the system detects >2 unique values in the Outcome / Prediction column, inject a new wizard step: The Cutoff Setter. The UI displays a histogram of their scores and asks: "Where is the cutoff line for approval?" The backend converts everything >X to 1 and <X to 0 before Fairlearn ever sees it.
Strategy 3: Safe-Math Wrappers (Fixes #6 - Division by Zero)
Implementation: Wrap all Fairlearn calculations in standard Exception handling arrays. If a denominator equals zero, the backend must return "value": null and a frontend explicit state "error": "Reference group recorded zero outcomes limit", completely preventing a backend crash.
Strategy 4: Hard File Limits & Chunking (Fixes #3 - The File Bomb)
Implementation: Set standard HTTP upload limits (MAX_UPLOAD_SIZE = 50MB). If a file is larger, reject it at the Nginx/FastAPI router level immediately. If it passes, use Pandas chunksize iterators so the file never loads into RAM entirely at once, protecting Google Cloud Run memory.
Strategy 5: Gemini Intelligent Caching (Fixes #15 - The Gemini Quota Wall)
Implementation: Add a hashing layer. When POST /api/report/generate is called, the backend takes an MD5 Hash of the audit_results.json. It checks the Postgres Database. If that hash exists, it instantly returns the cached Gemini text from the database. (This means during your hackathon demo running the same sample dataset over and over, you only ping the actual Gemini API once, guaranteeing no rate-limit crashes).
